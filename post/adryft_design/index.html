Adryft Conceptual Design

March 2021 5-minute read

Design Concept that addresses water-taxi boat interfaces and is built with Figma.

Duration: 3 week project for User Interface Design Class Role: UI Developer

Context

As part of ENP design assignment, I had to build a conceptual UI design for in-boat experiences and had an extension of Lyft branding, codenamed as Adryft. Adryft is Lyft’s new automous water taxis, and operates in markets where canals are a primary means of transportaion (e.g. Amsterdam, Venice).

Ideation

Design Considerations

The screens are in consideration that adryft is being used for one registered passenger at a time (may include a group under that one registered passenger). This way individuals do not collide in sharing the interface of the autonomous boat, like conflicting in who plays their music. This also allows personalization for each passenger. Map navigation only allows for designated stops (similar to bus and trains), which ensures the safety and maintenance of the boat. Ideally each stop has its own charging station and nearby help/maintenance booths. For the interface, the design prioritizes usability and quickness in navigation. Considering users may be using this as a daily commute (similar to bus and train navigation interface), the interface must be simple and intuitive to read.

  1. The interface is more of dashboard design than the usual navigation menu. This allows users to have access to everything on one screen rather than having to navigate many screens, which improves the simplicity of the interface. And so the main screen is divided into four main interactive sections: boat menu, feature selection buttons, feature selected information, and map interface.
  2. The use of icons, labels, and cards help readability. Important features such as: emergency, instructions, and voice assistant have their designated buttons on all screens. Whereas less important features such as selecting boat features are formed into cards helps prevent users from misclicking.
  3. Color palettes are composed of orange from the logo and gray scale colors. Simple colors help users associate what features hold greater importance. Color priority goes from orange, black, then gray, where orange is the most important color and gray is the least important.

Product Flow

Developing the Solution

Wireframes

Screens Descriptions
Briefing/Safety Briefing/safety includes information about warnings about using the autonomous boat, life jackets, location of first aid kit, instructions of what to do in case of emergency, and cautions users to practice safety (securing cargo, remain seated for the journey).
Help in using the screen Provides temporary labels over sections of the main menu.
Main Menu The boat menu includes greetings, trip parameters, button to receive help on emergency button, how to use interface, button to use voice assistant to interact with the boat, and displays alerts. Next is the feature menu selection, which comprises all the different settings users may want to adjust. This includes display settings, music connection, air configuration, temperature setting, and similar features to that of a car. By selecting one of the options, this affects the view of the feature selected menu. Feature selected menu is the view, users are allowed to expand for more information or continue using the condensed version of feature. Buttons per feature may vary. Map interface displays information regarding geographic location: highlighting landmarks during transit and enabling destination changes. Navigation is set to certain stops (similar to bus/train systems), since the boat can only steer to location with water passageways.

Moodboard

Finalization the Solution

Design feedback from the wireframes are updated onto the final mockups.

  1. For the boat menu for all screens, trip metrics are only shown when the boat is traveling state (after user types in address). A welcome message replaces the trip metrics on the other states/screens. Buttons are now labeled with “emergency”, “assistance”, and “voice assistant”. Assistance button was unclear before so icons were updated. The instructions screen is now accessible in the boat menu as the “assistance button”.
  2. For the instructions screen, button text is updated to “return” and includes an icon.
  3. For the condensed feature menu for all screens, the progress bar of the music is on top to help divide content later when the user wants to expand the feature menu.
  4. For the interacting with map part 2 screen, trip metrics text are labeled: arrival time, speed, time left, and miles left.

Reflection

This was an interesting problem space, since I do not own any smart motor vehicles, nor do I own a tablet, and there are no existing smart water taxi interfaces. A challenge I faced was researching the problem space as there are no current smart taxi boat interfaces, hence I based my interface primarily on Tesla’s smart car interface. There’s definitely a gap between the use of smart cars and the use smart water taxis. Since for smart water taxis, there is a great need for safety precaution since travel is over water, which makes the taxi sensitive to weather conditions. Also users are likely to be sensitive to the motion from boat, hence user may be focused more on balance then on the screen interface. For future iterations, running user study to determine usability would be useful in improving the design. Also building more screens for the prototype to help clarify user flow.
